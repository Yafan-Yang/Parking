# 项目新人上手指南

## 1. 这是什么项目

这是一个基于 Web 的停车场管理系统，面向三类角色使用：

- 管理员：负责系统配置、用户与员工管理、车位管理、公告发布、统计分析等
- 停车员工：负责一线停车业务处理，比如查看车位状态、处理预约、生成订单、跟进业务
- 普通用户：负责注册登录、维护车辆信息、预约车位、查看订单、支付停车费、购买会员

项目当前最核心的业务闭环是：

`查看车位 -> 发起预约 -> 生成停车订单 -> 驶出结算 -> 在线支付 -> 消息通知`

会员体系会插在支付链路里，对停车费用形成优惠联动。人脸识别、邮件、Redis 相关能力保留了接口或配置，但在当前版本里更偏“可选扩展”，不是主流程的强依赖。

## 2. 新人应该先怎么理解这个项目

如果你是第一次接手，建议不要先从所有菜单开始看，而是先抓住下面这条主线：

### 2.1 普通用户主流程

1. 用户登录系统
2. 在首页或预约页面查看当前空闲车位
3. 选择自己的车辆发起预约
4. 系统把对应车位状态改成“预约中”
5. 用户车辆入场后，员工或管理端生成停车订单
6. 系统把车位状态改成“停车中”
7. 用户驶出后，系统计算停车时长和停车费用
8. 用户完成在线支付
9. 系统更新订单状态，并发送站内消息提醒

### 2.2 会员相关流程

1. 用户选择会员规则
2. 系统生成会员订单
3. 支付成功后开通会员有效期
4. 会员有效期内，停车订单可按规则享受减免

### 2.3 管理端/员工端流程

1. 管理员维护车位、员工、公告、会员规则等基础数据
2. 员工查看当前车位状态和预约情况
3. 车辆入场时生成订单，车辆离场时进行结算
4. 管理端查看订单、收益、统计图表和运营信息

## 3. 关键业务状态

理解这个项目时，最重要的是先记住几个状态字段：

### 3.1 车位状态 `space_status_info.status`

- `-1`：预约中
- `0`：空闲
- `1`：停车中

### 3.2 预约状态 `reserve_info.status`

- `0`：结束
- `1`：预约中

### 3.3 停车订单状态 `park_order_info.status`

- `0`：未支付
- `1`：已支付

说明：

- 订单刚生成时也是 `0`
- 用户驶出后，如果还没支付，仍然是 `0`
- 支付完成后才会变成 `1`

### 3.4 当前版本的一个重要业务规则

- 预约默认有效期是 30 分钟
- 同一辆车不能同时存在“预约中”记录和“未完成订单”
- 用户驶出时系统会计算停车分钟数，并按车位单价换算总价
- 如果用户当前处于会员有效期内，停车订单可以被减免

## 4. 主要功能模块

### 4.1 停车场业务模块

这些是项目真正的核心业务功能：

- 用户管理：用户资料、联系方式、头像、账户关联
- 车辆管理：车辆档案、车牌号、车辆图片、所属用户
- 车位管理：车位名称、位置、图片、单价
- 车位状态管理：空闲、预约中、停车中
- 预约管理：查看可预约车位、提交预约、查看预约记录
- 订单管理：入场生成订单、离场结算、支付状态跟踪
- 支付管理：停车订单支付、会员购买支付
- 会员管理：会员规则、会员购买记录、会员有效期
- 消息通知：预约提醒、订单提醒、支付提醒、会员提醒
- 公告管理：停车场公告发布与展示
- 统计分析：收入统计、订单数量、趋势图、运营看板

### 4.2 框架自带或后台能力模块

这些模块更多是 FEBS 后台框架能力，停车业务会复用它们，但它们本身不是停车场主流程：

- 系统管理：用户、角色、菜单、部门、字典
- 监控管理：系统信息、Redis 信息、JVM、Tomcat、日志
- Quartz：定时任务与任务日志
- 个人中心：修改资料、头像、密码

如果新人只是想先看停车业务，不建议一开始把精力放在这些模块上。

## 5. 代码结构怎么找

项目整体是前后端分离结构：

- `backend/`：Java 后端
- `frontend/`：Vue 前端
- `backend/sql/`：数据库初始化脚本
- `Doc/`：项目说明文档
- `uploads/`：默认上传目录之一

### 5.1 后端重点目录

- `backend/src/main/java/cc/mrbird/febs/cos/`
  这里是停车场业务代码，建议新人优先看这个包

- `backend/src/main/java/cc/mrbird/febs/cos/controller/`
  业务接口入口，建议先看：
  - `ReserveInfoController`：预约逻辑
  - `ParkOrderInfoController`：订单生成、驶出结算、统计
  - `MemberRecordInfoController`：会员订单支付回调
  - `SpaceInfoController` / `SpaceStatusInfoController`：车位与状态
  - `PayController`：支付入口

- `backend/src/main/java/cc/mrbird/febs/system/`
  账号、角色、登录、菜单等通用后台能力

- `backend/src/main/resources/application.yml`
  后端端口、数据库、Redis、邮件、上传目录等配置都在这里

- `backend/src/main/resources/config/alipay.properties`
  支付宝相关配置

### 5.2 前端重点目录

- `frontend/src/views/user/`
  普通用户相关页面，比如预约、订单、会员、消息、车辆

- `frontend/src/views/manage/`
  管理端相关页面，比如车位、订单、公告、规则、员工、会员

- `frontend/src/views/staff/`
  停车员工相关页面

- `frontend/src/views/manage/component/work/Work.vue`
  普通用户首页里和停车主流程关系很强的页面，能看到车位状态、公告和发起预约逻辑

- `frontend/src/views/HomePage.vue`
  首页入口，不同角色会进入不同首页

- `frontend/src/utils/request.js`
  前端请求后端的统一入口，当前默认写死为 `http://127.0.0.1:9527/`

### 5.3 哪些代码最值得先读

如果你想在最短时间内看懂项目，推荐按下面顺序阅读：

1. `README.md`
2. `Doc/项目功能文档.md`
3. `backend/sql/init_parking_lot_cos_demo.sql`
4. `ReserveInfoController`
5. `ParkOrderInfoController`
6. `MemberRecordInfoController`
7. `frontend/src/views/manage/component/work/Work.vue`
8. `frontend/src/views/user/order/` 和 `frontend/src/views/user/reserve/`

## 7. 本地运行方式

### 7.1 运行前准备

建议准备：

- JDK 8
- Maven
- MySQL
- Node.js
- Yarn 或 npm

可选组件：

- Redis：不是当前主流程强依赖，但部分缓存、在线用户、Redis 监控相关能力会用到
- 邮件环境：用于发送邮件提醒，不配置也不影响主流程
- 支付宝沙箱：用于完整演示支付联调

### 7.2 第一步：初始化数据库

数据库名默认是：

`parking_lot_cos`

推荐首次使用完整演示脚本：

- `backend/sql/init_parking_lot_cos_demo.sql`

它会创建：

- 系统表
- 业务表
- 角色和菜单
- 默认账号
- 完整演示数据

如果你只想要最小可运行数据，可以用：

- `backend/sql/init_parking_lot_cos.sql`

### 导入示例

如果本地 MySQL 用户名密码都是 `root/root`，可以执行：

```bash
mysql -uroot -proot < backend/sql/init_parking_lot_cos_demo.sql
```

### 7.3 第二步：检查后端配置

后端配置文件：

- `backend/src/main/resources/application.yml`

当前默认关键配置如下：

- 后端端口：`9527`
- 数据库地址：`jdbc:mysql://127.0.0.1:3306/parking_lot_cos`
- 数据库账号：`root`
- 数据库密码：`root`
- 上传目录：`${FEBS_UPLOAD_DIR:uploads}`

如果你的本地数据库账号密码不同，先改这里再启动。

### 7.4 第三步：启动后端

方式一：IDE 启动

- 直接运行主类 `cc.mrbird.febs.FebsApplication`

方式二：命令行启动

```bash
cd backend
mvn spring-boot:run
```

后端启动成功后，默认接口地址是：

`http://127.0.0.1:9527`

### 7.5 第四步：启动前端

进入前端目录：

```bash
cd frontend
```

安装依赖：

```bash
yarn install
```

如果你不用 Yarn，也可以：

```bash
npm install
```

启动开发环境：

```bash
yarn run dev
```

前端默认端口是：

`http://localhost:8081`

登录页地址通常是：

`http://localhost:8081/login`

### 7.6 默认演示账号

如果你导入的是 `init_parking_lot_cos_demo.sql`，常用账号如下：

- 管理员：`admin / 123456`
- 普通用户：`fank / 123456`
- 停车员工：`staff01 / 123456`
- 其他演示用户：`user01 / 123456`、`user02 / 123456`

如果你导入的是 `init_parking_lot_cos.sql`，通常至少有：

- 管理员：`admin / 123456`
- 普通用户：`fank / 123456`

## 8. 运行时的几个重要注意点

### 8.1 前端把后端地址写死成了 `127.0.0.1:9527`

除了 `frontend/src/utils/request.js` 里的统一请求地址，很多上传和图片访问地址也直接写成了：

`http://127.0.0.1:9527`

所以如果你改了后端端口，或者要联调远端服务，不能只改一处，前端里这些地址也要一起调整。

### 8.2 上传文件目录和图片访问

后端通过 `/file/fileUpload` 上传文件，通过 `/imagesWeb/**` 暴露图片访问。

默认上传目录配置是：

`febs.file.upload-dir: uploads`

也就是说：

- 如果你按默认方式启动，上传文件通常会落在当前工作目录下的 `uploads/`
- 如果你想把上传目录固定到别的位置，可以设置环境变量 `FEBS_UPLOAD_DIR`

### 8.3 Redis 不是主流程强依赖，但不是完全没用

当前代码已经做了降级处理：

- Redis 不可用时，登录仍然可以继续
- Token 校验可以降级为纯 JWT 校验
- 在线用户、Redis 监控、部分缓存相关能力会受影响

所以新人第一次本地跑通项目时，可以先不把 Redis 当成硬门槛。

### 8.4 邮件和支付更像“环境依赖功能”

- 邮件能力依赖邮箱配置和网络环境
- 支付能力依赖支付宝沙箱配置
- 人脸识别依赖百度相关配置

这些功能可以保留，但不要把它们当成“第一次启动必须全部打通”的前置条件。

## 9. 支付、邮件、人脸识别怎么理解

### 9.1 支付

项目接了支付宝沙箱支付，相关配置在：

- `backend/src/main/resources/config/alipay.properties`

本地如果只是先熟悉业务流程，可以先把它当作联调型功能来看；如果要完整演示真实支付跳转，需要保证：

- 支付宝沙箱参数可用
- `notifyUrl` 和 `returnUrl` 可以正确访问

### 9.2 邮件

邮件配置在：

- `backend/src/main/resources/application.yml`

但当前版本邮件不是核心闭环的强依赖。即使邮件不可用，预约、订单、支付主流程仍然可以继续走。

### 9.3 人脸识别

人脸能力主要是展示和扩展接口，当前版本不强制绑定预约和支付闭环。

## 10. 新人第一次接手，建议这样跑和这样看

### 第一天建议目标

先完成这四件事就够了：

1. 用 demo SQL 跑起数据库
2. 启动后端和前端，成功进入登录页
3. 分别用管理员和普通用户账号登录，看清楚菜单差异
4. 用普通用户走一遍“查看车位 -> 预约 -> 订单 -> 支付”的流程

### 第二步建议阅读路径

跑通后再去看代码：

1. 看 `ReserveInfoController`，理解预约怎么改车位状态
2. 看 `ParkOrderInfoController`，理解订单怎么生成和结算
3. 看 `MemberRecordInfoController`，理解支付成功后的状态更新
4. 看前端 `Work.vue`、`user/reserve`、`user/order` 页面，理解页面如何调用接口

### 第三步再去看系统底层

最后再补：

- 登录鉴权
- 角色菜单
- 系统管理
- 监控与定时任务

这样理解效率会高很多。

## 11. 常见问题排查

### 11.1 前端能打开，但接口全报错

先检查：

- 后端是否已经启动
- 后端端口是不是 `9527`
- 数据库是否已成功导入
- 前端请求地址是不是仍然指向 `127.0.0.1:9527`

### 11.2 登录失败

先检查：

- 用的是不是 demo 账号
- 数据库导入的是不是正确脚本
- 数据库里 `t_user`、`t_user_role` 是否有数据

### 11.3 图片不显示

先检查：

- 上传文件是否真的落到了上传目录
- `/imagesWeb/**` 映射是否生效
- 前端图片地址是不是仍然指向 `http://127.0.0.1:9527/imagesWeb/...`

### 11.4 支付功能打不开或跳转异常

先检查：

- `alipay.properties` 是否可用
- 本地网络环境是否允许访问支付宝沙箱
- 回调地址和前端地址是否对应当前端口

## 12. 一句话总结

这个项目不是单纯的后台管理系统，而是一个围绕“车位预约、停车订单、结算支付、会员优惠、消息通知”展开的停车场业务系统。新人接手时，优先看 `cos` 业务包和用户侧主流程，先把链路跑通，再回头看系统管理和底层框架，会更容易进入状态。
